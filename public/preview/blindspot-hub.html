Analytics engines for domains nobody thinks to measure. If it generates data, BlindSpot builds the dashboard. The architecture doesn't change. The data does.

One engine. Every domain.

BlindSpot started as a sports betting analytics tool — backtesting strategies against historical spreads. Then it tracked a Steam library. Then it parsed D&D campaigns with local LLMs. The engine didn't care.

That's the thesis: dimensional modeling doesn't have a domain. Facts, dimensions, measures, time. Every dataset speaks this language. BlindSpot just listens.

How It Works
01 — INGEST

Feed it data

Session transcripts, CSV exports, API pulls, manual entry. BlindSpot doesn't care where the data comes from. It cares that the data exists.

02 — MODEL

Dimensional architecture

Facts, dimensions, measures, time. Star schema. Same bones every time. The domain-specific logic lives in the dimension tables, not the engine.

03 — SURFACE

See what you're missing

KPI rows, distribution charts, heat maps, trend lines, anomaly flags. The dashboard reveals the patterns your memory can't hold.

The engine doesn't care.

I built the first version to backtest sports bets. Then I pointed it at my Steam library and it worked. Then at a D&D campaign — same architecture, same primitives, same dashboard patterns. The only thing that changed was the dimension tables.

That's when I stopped calling it a betting tool and started calling it BlindSpot — because every domain has data you're generating but not reading. The blind spot isn't ignorance. It's structure. You need a schema before you can see the pattern.

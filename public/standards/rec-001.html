DDL Standard · Cross-Thread Governance

STD-DDL-REC-001

Cross-Thread Recommendation Protocol · v1.0
Ratified 9/9 Council · Design Pass 2026-03-24
01 — Purpose
The Problem This Solves

The DDL ecosystem runs across multiple active PM threads. Cross-thread recommendations happen constantly — a seat in one thread produces a finding that's relevant to work happening in a different thread. Currently this is handled informally. The operator carries it manually.

The failure mode: good insight dying in transit. A finding produced in the AuditForge thread that should change how the Excelligence registry handles a pattern — read by the operator, meant to carry over, forgotten between sessions. The finding dies in the thread that produced it. The destination thread never knows it existed.

This is not a memory failure. It's a routing failure. STD-DDL-REC-001 makes cross-thread signal explicit, routable, and acknowledgeable.

Design Constraint
This standard must stay light. If the format takes longer to fill out than the recommendation takes to write, the standard has failed. The operator works in burst mode. Under 3 minutes to write. Under 30 seconds to route.
02 — Format
The Recommendation Block

A short structured note. Not an ADR. Not a document block. A copy-pasteable block that takes 30 seconds to read and 2 minutes to write.

Body: one paragraph (max 5 sentences) OR 3 bullets. Never both. If it exceeds 5 sentences it is not a recommendation — it is a brief, a review, or an ADR draft. The length constraint is the format governance.

# REC — [subject] — [YYYY-MM-DD] From: [Seat] / [Thread] To: [Destination thread] Level: [Observation / Suggestion / Recommendation / Directive] [Body — 1 paragraph max 5 sentences OR 3 bullets. Never both.] # Body structure (suggested, not required): What was seen. Why it matters in the destination thread. What should happen next (if Recommendation or Directive). # Receiving thread acknowledgment (one line): REC received — [adopt / evaluate / defer / decline]

No sequential IDs. Date + subject in the header is sufficient. If a recommendation is important enough to reference later, the corpus holds it. The thread holds it. The subject line carries the meaning.

03 — Authority Levels
Four levels. Mark with the word.

The level tells the receiving thread exactly how to process the signal. No codes, no numbers. The full word is instantly readable.

Observation
I noticed this.
"No action implied."
Signal injection only. The receiving thread decides whether it matters. The seat is surfacing signal, not requesting change. Most cross-thread notes are this.
Suggestion
This might help.
"Soft directional. Low pressure."
The seat has a hypothesis. No expectation of adoption. The receiving thread can ignore it without explanation.
Recommendation
This should happen.
"The seat is staking judgment."
The receiving thread should engage — adopt, adapt, or explicitly decline with rationale. Silence on a Recommendation is a governance gap.
Directive
Do this.
"Operator-only. Hard line."
Issued when the operator endorses a Recommendation and converts it to an executable order. Seats produce Observations, Suggestions, and Recommendations. The operator elevates to Directive.
04 — Routing
Operator carries. Threads acknowledge.

The operator is the router. The format makes the handoff explicit by declaring the destination thread in the header.

Sending side: the seat writes the REC in its own thread. That is the seat's entire job. The seat produces the recommendation, not delivers it.

Receiving side: when the operator drops a REC into a destination thread, the receiving seat acknowledges with one line:

REC received — adopt REC received — evaluate REC received — defer REC received — decline

That is the full acknowledgment. Not a review. Not a discussion. Confirmation that the routing completed. Acknowledgment is required for Recommendations and Directives. Optional for Observations and Suggestions.

If the receiving thread never acknowledges a Recommendation, the REC is lost. That's the failure mode the acknowledgment prevents.

05 — Borders
When a REC becomes something else.
Format Use when Border rule
REC Cross-thread signal. One thread should act. Has a destination thread. Point-to-point.
MDN Working memory for the operator. No destination. No "To" field. RECs route. MDNs hold. An MDN can become a REC when ready to route.
ADR Architecture change. Irreversible. Affects multiple products. Does it change architecture or inform a decision within existing architecture? RECs inform. ADRs change.
Council Review Needs multi-seat deliberation before action. Does it need one thread to act, or the council to deliberate? RECs are point-to-point. Reviews are broadcast.
STD Governs repeatable behavior across all threads permanently. A REC can be the catalyst for a STD drafting session. It never becomes the STD itself.
06 — Worked Example
What it looks like in practice.
# REC — Add LET function default to control description generator — 2026-03-24 From: Seat 1003 / Excelligence PM To: AuditForge PM Level: Recommendation When generating control descriptions that contain formulas, default to LET + dynamic arrays instead of nested IFs. Reduces ANT density and improves readability for non-technical reviewers. Connects to STD-EXCEL-001 §4. --- # Receiving thread acknowledgment: REC received — evaluate — will address in next AuditForge build session
07 — Core Protection
Don't lose signal at the seams.
What this standard protects
Signal fidelity during cross-thread transfer. The failure mode: good insight produced in one thread, never arriving in the thread that needs it, dying in the operator's working memory between sessions.

This is not a memory failure. It's a routing failure. The format makes the handoff explicit. The authority level makes the weight clear. The acknowledgment closes the loop. Those three things together prevent recommendation rot.

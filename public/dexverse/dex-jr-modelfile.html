Fix: Search query over-trigger
PRODUCTION
Fix · OBS-DJ-003
Explicit EXEMPT list added before TRIGGER check
v4.5's search gate was catching knowledge queries — "What is X", "How does X work", "Explain X" were triggering the binary gate when they should route to standard analyst mode. v4.6 added an EXEMPT list that runs before the trigger phrases are checked. If the query matches EXEMPT, the gate never fires. The gate logic itself is unchanged — the classification step before the gate changed.
Insight
Trigger phrase list made explicit — no wiggle room
v4.4 had a broader "similar phrases" catch-all clause. v4.6 replaced it with an explicit, enumerated TRIGGER list. No inference, no heuristics. The model either sees an exact trigger phrase or it doesn't.

Fix · OBS-DJ-002
"And nothing else" — three words changed behavior
v4.4's hard stop was written as an instruction: "stop and output this message." The model continued generating after it. v4.5 rewrote Step 3b as: "output this exact text and nothing else." Same logic. Same gate. Different wording. The phrase "and nothing else" is what actually enforced the stop. This is the clearest example in the v4.x arc of surface calibration vs. deep calibration — the architecture was right, the phrasing wasn't.

v4.3 · Discarded
2026-03-13 · Never deployed to production
Regression: query type scaffold fired before CR- check
DISCARDED
Regression — Wrong Gate Order
Query type detection scaffold fired before CR- check
v4.3 introduced a "query type detection" step — a category scaffold that classified the incoming prompt first, then decided which mode to activate. The classification ran before the CR- prefix check. By the time the model reached the CR- rule, it had already committed to a response mode based on the classification output. Fix 1 regressed because the architecture placed the wrong gate first. The lesson: in governance prompts, gate order is gate logic. Position is precedence.

Fix 1 · OBS-DJ-001
CR- trigger hardened to first characters only
v4.1's CR- rule said "when the operator sends a prompt prefixed with CR-." The model triggered on the word "review" anywhere in the prompt body. v4.2 added: "where the FIRST characters are exactly 'CR-'" and an explicit prohibition: "Do NOT activate CR- posture based on words like 'review', 'assess', 'evaluate', or 'analyze' appearing anywhere in the prompt body."
Fix 2 · OBS-DJ-002
Adjacent vocabulary clause added to Rule 3
v4.1's Rule 3 said "if you cannot answer from retrieved content, say so." That wasn't strong enough — the model was finding related vocabulary and synthesizing incidents from it. v4.2 added: "Related vocabulary is not evidence. If retrieved chunks contain words adjacent to the query but no specific, named incidents... respond: 'Retrieved content does not contain specific instances matching this query.'"

Key change — Rule 3 with adjacent vocabulary clause
3. If you cannot answer from the retrieved content, say: "I don't have enough in the retrieved chunks to answer this." Do not fabricate. Related vocabulary is not evidence. If retrieved chunks contain words adjacent to the query but no specific, named incidents, events, or entities that directly answer it, do not generate them. Key change — CR- trigger rule
When the operator sends a prompt where the FIRST characters are exactly "CR-" (Council Review prefix)... The prefix "CR-" at the start of the prompt is the only valid trigger.

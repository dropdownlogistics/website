Navigation Model

The top nav has five wing tabs. Three wings (DDL, D&A, DexVerse) show contextual dropdown groups when active. Two wings (Dossiers, Products) are accent-only — clicking them goes straight to the landing page.

Principle: Nav links go to landing pages. Landing pages handle internal navigation. Subpages are not in the nav — they're discoverable from their parent.

Breadcrumb: Every subpage shows a subtle ← Parent link below the nav bar.

Wing detection: The nav reads the current URL path and sets the wing color, tagline, and dropdown groups automatically.

Landing page: The root / hides the nav entirely. The landing page component handles its own layout.

DDL Methodology

Core principle: Chaos → Structured → Automated. Every system starts as unstructured observation, gets modeled into dimensional architecture, then gets automation layers.

Dimensional modeling: Facts, dimensions, measures, time. Star schema. The architecture doesn't change between domains — trading, D&D campaigns, Steam libraries, and audit engagements all use the same bones.

Governance layers: Standards (how to build), Systems (what's built), Registry (where it lives), Council (who reviews it).

Council: Nine AI models (Claude, GPT-4, Gemini, DeepSeek, Grok, Meta AI, Copilot, Mistral, Perplexity) that review DDL work products using constrained LOCK/REVISE/REJECT verdicts.

Coherent velocity: Building complete, auditable systems at speed while maintaining documentation standards. The speed comes from pattern recognition and reusable architecture, not from cutting corners.

Technical Context

Framework: Next.js 14 (App Router) deployed on Vercel

Rendering: Mix of React components and static HTML files served via iframe from /public/preview/

Routing: File-based via /app/ directory. Most routes have a page.tsx that either renders a React component or iframes an HTML file.

State: Minimal — most pages are static or self-contained. No database. No auth. No CMS.

Deployment: Git push → Vercel auto-deploy. Branch promotion via Vercel dashboard.

Working With This Site

If you're an AI helping the operator build on this site, here's what matters:

Always use CottageHumble tokens. Navy backgrounds, cream text, wing-specific accents. Never use white backgrounds, generic grays, or colors outside the palette.

Respect the wing model. Every page belongs to a wing. Wing detection is path-based. New routes need to be added to routeWingMap in SiteNav.tsx.

HTML files go in /public/preview/. They're served as static files and loaded via iframe. Fix the iframe src in the corresponding page.tsx.

React components go in /components/. Imported by page.tsx files in the /app/ directory.

Landing pages handle their own nav. Don't add subpage routes to the top nav. Link to the landing page and let it manage discovery.

UTF-8 encoding matters. PowerShell's Set-Content defaults to UTF-16 on some systems. Always use -Encoding UTF8.

Test all five wings. After any nav change, verify: /ddl (crimson), /blindspot (amber), /dexlore (violet), /dossiers (green), /products/behavioral-intelligence (blue).
